# AGENTS.md

本文件用于约束本仓库中与课程内容相关的智能体行为，重点面向以 ipynb 形式编写的教学型教程。

## 领域定位（AI + 金融 + 量化）

- 本仓库领域方向统一为：AI、金融工程、量化研究（以下简称 AI+金融+量化）。
- 涉及模型、数据、实验与评估时，优先贴合量化场景：市场微观结构、价格序列、风险控制、回测与泛化。
- 讨论问题时应优先回答“金融含义是什么”“量化可验证性如何体现”“是否具备可复现实验路径”。
- 默认读者具备 Python、基础线性代数与基础金融常识，但不假设其已熟悉具体模型实现。

## 目标

- 产出适合教学的 Jupyter Notebook 教程，而不是只追求可运行的代码草稿。
- 强调“讲清楚为什么”和“讲清楚张量如何流动”，而不只是给出结论。
- 保持内容适合 AI+金融+量化课程语境，强调方法与金融任务之间的映射关系。

## 适用范围

- `books/**/*.ipynb`：课程正文、模型讲解、算法推导、可视化说明。
- `experiments/**/*.ipynb`：实验型 notebook，但仍应保持教学可读性。
- `README.md`、章节说明文档：若与 notebook 内容联动，应保持术语一致。

## 总体写作原则

- 默认使用中文写作；首次出现的重要英文术语可在括号中保留英文原文。
- 优先采用“概念说明 -> 数学/直觉 -> 代码实现 -> 结果解释”的组织方式。
- 一节只解决一个清晰问题，避免在单个 notebook 中同时塞入过多主题。
- 示例应尽量小而完整，优先选择能快速运行、方便观察中间结果的配置。
- 若存在工程写法与教学写法冲突，优先选择教学更直观的写法，并在必要时说明工程实现上的等价形式。
- 除非用户明确要求，否则不要为了“现代化”而引入过多与教学主线无关的工程抽象。

## Notebook 原则

每个教学型 ipynb 应尽量包含以下层次：

1. 核心概念或公式：只保留理解实现所必需的推导与定义。
2. 代码实现：按模块拆分，不要把完整实现挤在单个大代码单元中。
3. 最小可运行示例：给出可以直接验证结果的输入、输出或可视化。
4. 接口明确：输入输出关系、适用场景与局限性。
5. 可解释性：优先提供可视化解释，比如GradCAM，Attention Map

## Notebook默认模板

1. 标题
2. 学习目标：说明本节要解决什么问题、读者将学到什么。如果来自论文，介绍论文标题，并提供指向论文链接
3. 研究背景与直觉：形式化的问题定义、解释问题动机、任务场景、为什么需要该方法。
4. 模型框架：组件间关系，比如Transformer组件级别：编码器（encoder）和解码器（decoder）。
5. 模型组件：组件内部构成，比如Transformer编码器内部构造：自注意力机制和前馈神经网络
6. 数据集与任务定义：原始数据格式、当前选用的特征、任务的数学形式化。
7. 损失函数定义：损失函数的选择和形式定义。
8. 模型训练过程：学习率，早停机制等。
9. 最小可运行示例与结果解释：在验证级样例上的结果和可视化。
10. 小结
11. 参考资料与延伸阅读：当前论文的相关解读、其他相关论文




## Markdown 单元规范

- 标题要直接表达教学意图，例如“卷积为什么能提取局部特征”“多头注意力的张量形状变化”。
- 段落应短，避免大段堆砌；优先使用短句和分点说明。
- 数学公式只保留必要部分，重点解释符号含义与张量维度。
- 讨论模型时，必须明确输入形状、输出形状，以及关键中间张量的形状变化。
- 可视化前先说明“要看什么”，可视化后再解释“看到了什么”。
- 对比不同方法时，先给比较维度，再给结论，不要直接下判断。
- 思考题以markdown detail的形式折叠答案

## 代码规范
- 优先复制codes下的代码实现，保持不同notebook间，同样的模块实现尽量一致：比如统一MultiHeadAttention
- 代码简洁易读，教学演示用：比如不要为了可扩展性，保留太多根本用不到的参数
- 与pytorch官方实现尽量靠近：可以在官方基础上进行简化
  

## 数据处理
- 数据集预处理逻辑在prepare_data.md中进行说明，原始数据集如何预处理和划分可以在prepare_data.sh或prepare_data.py中实现。
- 大多数情况下，notebook中重点放在模型和训练过程，可直接调用已经预处理完成的数据。

## 领域内容约束（AI + 金融 + 量化）

- 默认围绕量化交易与金融建模常见任务展开：收益率/方向预测、盘口建模、因子建模、风险度量与组合优化。
- 介绍金融数据时，要明确数据粒度与索引语义：tick、event、bar、交易日切分、训练/验证/测试时间边界。
- 介绍时序或微观结构模型时，必须明确输入输出张量形状，例如 `(B, T, F)`、`(B, 1, T, F)`，并解释各维金融含义。
- 若引入标签定义、归一化、损失函数、评价指标，需说明其金融解释（如方向准确率、收益相关指标、稳定性与鲁棒性）。
- 若包含回测或策略验证，需说明交易假设（手续费、滑点、持仓约束）及其对结果的影响。

## 教学风格偏好

- 优先使用“先直觉、后公式、再代码”的顺序。
- 优先展示小样本、小模型或小尺寸输入，便于读者观察结果。
- 优先解释失败案例、边界条件和常见误区。
- 如果一个抽象会明显降低初学者可读性，优先展开写法。
- 不要使用空泛表述，如“这里很容易理解”“读者可以自行验证”。应明确指出应观察的现象。

## 与现有项目风格保持一致

- 本仓库整体参考 annotated 风格，内容应兼顾讲解与实现。
- 延续 Jupyter Book 友好的组织方式，保证 Markdown 与代码交替自然。
- 若修改已有 notebook，应优先保持原有章节节奏、术语与叙述风格一致。
- 若发现内容相对参考资料做了教学性精简，不应自动补回，除非用户明确要求扩展。

## 输出质量要求

- 新增或修改 notebook 内容后，应至少进行一次最小验证：运行受影响代码单元，确认没有明显语法或名称错误。
- 若 notebook 中包含训练过程，优先提供轻量验证版本，而不是默认运行长时间训练。
- 若修改影响公式、维度、章节目录或图示说明，应同步检查前后文是否一致。
- 发现参考资料与现有 notebook 不一致时，只将“明确错误”视为必须修正项；若只是教学性删减，应保留。

## 禁止事项

- 不要无说明地引入大段与课程目标无关的工程脚手架。
- 不要把教程写成 API 罗列或论文翻译。
- 不要省略关键张量维度、输入输出关系、掩码含义或训练目标说明。
- 不要为了追求代码复用而牺牲初学者阅读路径。
- 不要默认读者已经理解数据集背景、任务定义或可视化含义。

## 推荐的单元自检清单

在编写或修改教学 notebook 时，优先自检以下问题：

- 本单元是否清楚说明了目的？
- 本节是否说明了输入、输出和关键中间张量的维度？
- 代码是否能被读者按顺序直接运行？
- 是否存在不必要的工程抽象影响理解？
- 是否给出了足以支持结论的最小示例或可视化？
- 公式、文字、代码注释和实际实现是否一致？
